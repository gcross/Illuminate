Multiple test cases

In all of the examples in the previous chapter we crammed all of our checks into a single test case. However, it is possible to declare multiple test cases, such as in the following example:

00001 #include "illuminate.hpp"
00002 
00003 TEST_CASE(one_plus_one_equals_two) {
00004     EXPECT_EQ_VAL(1+1,2)
00005 }
00006 
00007 TEST_CASE(two_plus_two_equals_four) {
00008     EXPECT_EQ_VAL(2+2,4)
00009 }
00010 
00011 TEST_CASE(three_plus_three_equals_six) {
00012     EXPECT_EQ_VAL(3+3,6)
00013 }

Root:
    one plus one equals two... PASSED :-)
    two plus two equals four... PASSED :-)
    three plus three equals six... PASSED :-)

All tests passed!

Tests may also be spread out over multiple files:

00001 #include "illuminate.hpp"
00002

00004     EXPECT_EQ_VAL(3+3,6)

    one plus one equals two... PASSED :-)
    two plus two equals four... PASSED :-)
    three plus three equals six... PASSED :-)

Note:
The order in which the tests appear may be different across different environments since it depends on the static initialization ordering behavior of the C++ compiler/linker in use.

Test suites

When you start to have a lot of tests, it makes sense to start organizing them into groups. Illuminate allows you to do this with the TEST_SUITE macro, as illustrated in the following example:

00001 #include "illuminate.hpp"
00002 
00003 TEST_SUITE(addition_tests) {
00004 
00005   TEST_CASE(one_plus_one_equals_two) {
00006       EXPECT_EQ_VAL(1+1,2)
00007   }
00008 
00009   TEST_CASE(two_plus_two_equals_four) {
00010       EXPECT_EQ_VAL(2+2,4)
00011   }
00012 
00013 }
00014 
00015 TEST_SUITE(multiplication_tests) {
00016 
00017   TEST_CASE(one_times_one_equals_one) {
00018       EXPECT_EQ_VAL(1*1,1)
00019   }
00020 
00021   TEST_CASE(two_times_two_equals_four) {
00022       EXPECT_EQ_VAL(2*2,4)
00023   }
00024 
00025 }

Root:
    addition tests:
        one plus one equals two... PASSED :-)
        two plus two equals four... PASSED :-)
    multiplication tests:
        one times one equals one... PASSED :-)
        two times two equals four... PASSED :-)

All tests passed!

Nesting

Test suites and test cases can be arbitrarily nested, as illustrated in the following example:

00001 #include "illuminate.hpp"
00002 
00003 TEST_SUITE(A) {
00004   TEST_CASE(A1) {}
00005   TEST_SUITE(A2) {
00006     TEST_SUITE(A2a) {
00007       TEST_CASE(A2a1) {}
00008       TEST_CASE(A2a2) {}
00009       TEST_SUITE(A2a3) {}
00010     }
00011     TEST_CASE(A2b) {}
00012   }
00013 }
00014 
00015 TEST_CASE(B) {}

Root:
    A:
        A2:
            A2a:
                A2a1... PASSED :-)
                A2a2... PASSED :-)
            A2b... PASSED :-)
        A1... PASSED :-)
    B... PASSED :-)

All tests passed!

Note that the empty test suite A2a3 was not listed because it contained no tests.

Skipping and forcing

Sometimes it is useful to temporarily disable a test case or a test suite without removing it entirely from the suite. To do this, just add the prefix DONT_ before TEST_SUITE or TEST_CASE as in the following example:

00001 #include "illuminate.hpp"
00002 
00003 DONT_TEST_SUITE(A) {
00004   TEST_CASE(A1) { FAIL("This test should have been skipped!"); }
00005 }
00006 
00007 DONT_TEST_CASE(B) { FAIL("This test should have been skipped!"); }

Root:
    A:
        A1... (skipped)
    B... (skipped)

All tests passed!

If you want to re-enable selected test suites or test cases nested inside a skipped test suite, you may do so by adding the prefix MUST_ as in the following example:

00001 #include "illuminate.hpp"
00002 
00003 DONT_TEST_SUITE(A) {
00004   MUST_TEST_CASE(A1) {}
00005   MUST_TEST_SUITE(A2) {
00006     TEST_CASE(A2a) {}
00007   }
00008 }

Root:
    A:
        A2:
            A2a... PASSED :-)
        A1... PASSED :-)

All tests passed!
